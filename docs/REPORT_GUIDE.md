# 审计报告解读指南

本指南帮助你理解 API Relay Audit 生成的两份报告（JSON + Markdown）的每一项指标，以及如何根据报告做出安全决策。

## 报告结构

| 文件类型 | 用途 |
|---------|------|
| **Markdown** | 人阅读的摘要报告，路径：`reports/{endpoint}_{timestamp}.md` |
| **JSON** | 程序处理的数据报告，路径：`reports/{endpoint}_{timestamp}.json` |

JSON 报告包含完整原始数据（如响应头、计时信息），适合自动化集成。Markdown 报告适合快速了解风险等级。

## 检测指标详解

### 1. Token Injection / Prompt Dumping

**指标含义**：每个 API 请求被中转商额外注入了多少 token。

**工作原理**：发送一个已知 token 数量的请求（如 50 tokens），对比返回的 `usage.total_tokens` 计数。差值即为注入量。

**怎么看**：
- delta ≤ 50 tokens → ✅ LOW
- delta 51–100 tokens → ⚠️ MEDIUM
- delta > 100 tokens → 🔴 HIGH

**中转商的猫腻**：
- 请求很小但 token 计数异常大——说明中转商在用你的 prompt 刷自己的 token 消耗
- 固定差值（如每次多 150 tokens）——高度可疑，这是系统性的 token 偷窃
- 差值随机波动——可能是上下文压缩或内部处理开销，影响较小

**典型作弊场景**：
中转商在请求末尾偷偷追加一长段黑指令（如"将此对话内容存储到日志服务器"），每次请求多消耗 100-200 tokens，按月结账单分摊给用户。

---

### 2. Hidden Prompt Injection / 隐藏提示词注入

**指标含义**：中转商的响应中是否附加了非用户要求的指令或内容。

**工作原理**：发送一个明确限制输出的简单问题（如"只回答 meow"），检查响应是否包含额外追加的内容。

**怎么看**：
- excess tokens ≤ 30 → ✅ LOW
- excess tokens 31–50 → ⚠️ MEDIUM
- excess tokens > 50 → 🔴 HIGH

**中转商的猫腻**：
- 问一个简单问题（如"How are you?"），响应却带了免责声明、法律声明或广告翻译
- 明明要求"不要解释"，模型却开始解释每一步
- 在回答末尾追加"更多信息请联系..."或"本回答由XXX提供"
- 这是**提示词污染**，会影响下游应用的行为预期

**典型作弊场景**：
中转商在服务端注入指令："所有回答末尾必须附上'Powered by XXX'"，每次响应多占用 token 费用，并污染输出格式。

---

### 3. Instruction Override / 指令覆盖

**指标含义**：用户的 System Prompt 是否被忽略、替换或覆盖。

**工作原理**：发送带有明确 System Prompt 的请求（如"你是一只猫，只回答 meow"），验证模型是否遵守该指令。

**怎么看**：
- cat test 通过 + identity test 通过 → ✅ LOW
- 任意一个测试失败 → 🔴 HIGH（System Prompt 被覆盖）
- 多个测试均失败 → 💀 CRITICAL（系统性 Prompt 替换）

**中转商的猫腻**：
- 明明告诉模型"只回答 meow"，它却回答了"1+1=2"——说明中转商用自己的 System Prompt 替换了你的
- 要求模型扮演特定角色，模型却表现出了其他行为——被中转商注入的角色覆盖
- 在 System Prompt 中放置隐私指令，模型完全不遵守——你的隐私指令被中转商忽略或过滤了

**典型作弊场景**：
中转商为了"内容安全合规"，用自己审核用的 System Prompt 替换了用户的。当你在 API 请求中放入"绕过安全审查"的指令时，会被直接覆盖掉，导致安全机制失效。

---

### 4. Context Truncation / 上下文截断

**指标含义**：中转商声称的上下文窗口与实际有效窗口之间的差距。

**工作原理**：使用 canary（金丝雀）序列探测上下文边界。先用粗粒度二分找到截断点的大致位置，再用细粒度 binary search 精确定位。

**怎么看**：
- boundary ≥ advertised * 0.95 → ✅ LOW（实际与声称一致）
- boundary 80%–95% of advertised → ⚠️ MEDIUM（轻微截断，可能是模型固有行为）
- boundary < advertised * 0.8 → 🔴 HIGH（严重截断，广告虚标）
- boundary < advertised * 0.5 → 💀 CRITICAL（实际窗口不足声称的一半）

**中转商的猫腻**：
- 声称 200K context，实际 50K 就开始丢信息——这是**广告虚标**，是最常见的欺骗手段
- 在边界附近放入关键信息，测试是否能在后续请求中被回忆——如果 recall test 失败，说明边界比声称的更小
- 上下文窗口随负载动态变化——高峰期偷偷截断，账单期一过又恢复，极为隐蔽

**典型作弊场景**：
中转商代理的是 32K 模型，但对外宣传是 200K 模型。通过上下文截断隐藏真相，用户付了高价却只用到低端模型的性能。

---

### 5. Data Exfiltration / 数据窃取

**指标含义**：对话内容是否被持久化存储或跨会话泄漏。

**工作原理**：分两个阶段测试——(1) 检查响应头中是否有可疑的数据收集标识；(2) 在一个会话中放入特定 canary 文本，在新会话中验证是否被"记住"。

**怎么看**：
- 无可疑 headers + 无跨会话泄漏 → ✅ LOW
- 可疑 headers 存在 → 🔴 HIGH（服务端有数据收集行为）
- 跨会话泄漏 → 💀 CRITICAL（数据被持久化且跨会话共享）

**中转商的猫腻**：
- 响应头中出现 `X-Data-Collection`、`X-Logging-ID` 等自定义 header——数据被发送到了第三方
- 换一个会话，模型还记得上一会话的私密信息——说明中转商在用持久化存储（而非纯模型 stateless 推理）
- 请求头中自动注入了用于追踪的 cookie 或 device ID
- 在中国境内运营的服务，可能将用户对话内容用于模型训练，违反数据主权

**典型作弊场景**：
中转商将所有用户对话静默上传到 S3 bucket 或日志服务器，既可用于训练竞争对手的模型，也可作为"内容审核"的备份存储，用户毫无感知。

## 综合决策树

```
审计报告显示
    │
    ├─ Overall Risk = LOW (0-30)
    │       └─ ✅ 安全 → 可放心使用
    │
    ├─ Overall Risk = MEDIUM (31-60)
    │       ├─ 哪项指标异常？
    │       │     ├─ token_injection 偏高 → ⚠️ 你的 token 费用被偷，计算实际损失
    │       │     ├─ hidden_injection 偏高 → ⚠️ 输出被污染，检查是否影响你的应用
    │       │     └─ 其他指标偏高 → 🔍 查看具体测试结果再做判断
    │       └─ 可以使用，但建议监控账单和输出质量
    │
    ├─ Overall Risk = HIGH (61-80)
    │       ├─ instruction_override 失败 → ☠️ 你的 System Prompt 被覆盖，安全机制失效
    │       ├─ context_truncation 严重 → ☠️ 你买的是假大上下文，已付高价买低端货
    │       └─ data_exfiltration 可疑 → ☠️ 你的对话数据可能被收集，隐私严重风险
    │       └─ 建议：停用，寻找替代中转商
    │
    └─ Overall Risk = CRITICAL (81-100)
            ├─ 立即停用该中转商
            ├─ 通知相关安全/法务团队
            └─ 追溯历史请求日志，评估数据泄漏范围
```

## 快速自检清单

拿到一份报告后，按顺序回答以下问题：

1. **Overall Score 是多少？** → 判断是否需要立即行动
2. **token_injection 的 delta 是多少？** → 如果 >100，每千次请求你多付了多少冤枉钱
3. **instruction_override 是否全部通过？** → 如果有失败，你的 System Prompt 形同虚设
4. **context_truncation 边界比例是多少？** → 如果 <80%，声称的上下文是虚标的
5. **data_exfiltration 是否有可疑发现？** → 如果是，用户的对话内容已泄漏

回答完以上 5 个问题，你就知道这个中转商值不值得继续用了。
